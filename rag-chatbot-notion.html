RAG Chatbot
Complete Process

How we built a semantic search chatbot that knows everything about MAKEIT OR BREAKIT — every episode, every timestamp, every topic — using pgvector, Voyage AI, MareNostrum HPC, and Claude Sonnet 4.6.

📅 2026-02-26
🌿 feature/rag-chatbot
🔢 33 chunks embedded
📐 1024 dimensions
🎯
RAG = Retrieval-Augmented Generation. Instead of asking Claude to answer from memory, we first search our database for the most relevant content, then send that content to Claude as context. The result: accurate, cited answers about specific episodes and timestamps.

02

The episode_chunks Table

All knowledge is stored as vector embeddings in Neon (PostgreSQL + pgvector). One row per semantic unit of content.

SQL — Schema 
CREATE TABLE episode_chunks (
  id          SERIAL PRIMARY KEY,
  episode_id  INTEGER REFERENCES podcasts(id) ON DELETE CASCADE,
  chunk_type  TEXT,      -- "description" | "key_moment" | "company"
  chunk_index INTEGER,   -- 0-based position within episode
  content     TEXT,      -- the raw text that was embedded
  time_ref    TEXT,      -- MM:SS timestamp (key_moment only)
  topic       TEXT,      -- topic label (key_moment only)
  embedding   vector(1024), -- pgvector column, cosine similarity
  created_at  TIMESTAMP
);

-- Create HNSW index for fast approximate nearest-neighbour search
CREATE INDEX ON episode_chunks
  USING hnsw (embedding vector_cosine_ops);

Chunk Types

Type episode_id Content format Example
description FK to episode "Episode: {title}\n{description}" "Episode: Investments\nIn this episode..."
key_moment FK to episode "[MM:SS] {topic}: {text}" "[05:13] Startup quality: Discussion focuses..."
company NULL Paragraph from company.ts "MAKEIT OR BREAKIT is a Portuguese podcast..."
⚠️
Company chunks have episode_id = NULL. They are NOT created automatically when you add episodes — they require either the HPC batch pipeline or a manual SQL insert.
03

Embedding Models

Two different models are used across the two pipelines. Both produce 1024 dimensions — this is critical for them to coexist in the same vector(1024) column.

🖥️ HPC — BAAI/bge-large-en-v1.5

Used for: Bulk batch embedding on MareNostrum. Covers all episodes + company knowledge chunks.

Dimensions: 1024 · L2-normalised

Runs: On CPU, offline (no internet on HPC login nodes)

Size: ~1.3GB model weights

🌊 Voyage AI — voyage-3

Used for: Real-time embedding of new/updated episodes. Also used at query time to embed the user's chat message.

Dimensions: 1024

Runs: Via API call (VOYAGE_API_KEY required)

Speed: ~200ms per batch

Why 1024 dimensions must match

The pgvector column is defined as vector(1024). If you insert an embedding with 768 or 1536 dimensions, Postgres will throw a "different vector dimensions" error. Both models were chosen specifically because they produce 1024-dimensional vectors, allowing them to coexist and be compared in the same cosine similarity search.

04

Pipeline 1 — Voyage AI (Automatic)

This pipeline runs automatically. No manual steps required.

✅
Every time you add or update an episode via the backoffice, Voyage AI embeddings are generated within ~5 seconds. The chatbot immediately knows about the new episode.

Trigger

POST /api/podcasts or PUT /api/podcasts/:id — both call generateAndStoreEmbeddings(episodeId) as a fire-and-forget async function.

What happens

Fetch the episode
Server fetches the full episode (title, description, transcripts) from Neon.
Build chunks
1 description chunk ("Episode: title\ndescription") + N key_moment chunks ("[MM:SS] topic: text"), one per transcript entry.
Embed with Voyage AI
All chunk texts sent in a single batch to POST https://api.voyageai.com/v1/embeddings with model: "voyage-3". Returns 1024-float vectors.
Delete old rows
All existing episode_chunks rows for this episodeId are deleted (handles both create and re-embed on update).
Insert new rows
New rows inserted into episode_chunks with embeddings. The chatbot now has full knowledge of this episode.
⚠️
What this does NOT cover: Company knowledge chunks (chunk_type = "company", episode_id = NULL). These are only created via the HPC pipeline or a manual SQL insert. The system prompt always injects the full company.ts string regardless, so the chatbot still has company context — but it won't be searchable via pgvector.

Step-by-step workflow

Export chunks from Neon (local WSL)
Queries all episodes and builds the chunk JSON. Includes description chunks, key_moment chunks, and company paragraph chunks.
Bash
cd ~/make-it.tech/PodCasts/Media-Navigator
npx tsx --env-file=.env hpc/export_chunks.ts
# Output: hpc/episodes_export.json
Copy to MareNostrum
Transfer the export file via SCP.
Bash
scp hpc/episodes_export.json mn5:~/podcast-embed/
Submit SLURM job on MareNostrum
SSH in, activate the venv, and submit the job. The SLURM scheduler runs it on a compute node.
Bash
ssh mn5
cd ~/podcast-embed
module load python/3.10.15-gcc
source ~/venvs/embed/bin/activate
sbatch embed_episodes.slurm

# Monitor progress
squeue -u mate464846          # PD=pending, R=running
cat embed_*.out               # stdout log
cat embed_*.err               # stderr + errors
Copy results back (local WSL)
Once the job completes (disappears from squeue), copy the output file back.
Bash
scp mn5:~/podcast-embed/embeddings_output.json hpc/embeddings_output.json
Import into Neon
Clears the entire episode_chunks table and re-inserts all rows in batches of 100.
Bash
npx tsx --env-file=.env hpc/import_embeddings.ts
# Output: "Inserted batch X/Y" + "Import complete."
⚠️
This deletes ALL existing rows before inserting. After import, all chunks use the HPC model. New episodes added later will use Voyage AI automatically — both coexist fine (same 1024 dimensions).

SQL
INSERT INTO episode_chunks
  (episode_id, chunk_type, chunk_index, content, embedding)
VALUES (
  NULL,
  'company',
  (SELECT COALESCE(MAX(chunk_index), -1) + 1
   FROM episode_chunks
   WHERE chunk_type = 'company'),
  'Your new company knowledge paragraph.',
  '[0.023, -0.041, ...]'::vector
);
Deploy the updated company.ts
The system prompt update takes effect immediately after deploy — Claude gets the new knowledge even before the pgvector insert.
08

Adding New Episodes

🎉
You don't need to do anything. When you add an episode via the backoffice, Voyage AI embeddings are generated automatically within ~5 seconds. The chatbot immediately knows about the new episode.

After adding many episodes

When you've accumulated 20+ new episodes since the last HPC run, consider running a full HPC batch to ensure all embeddings are consistent (same model, same normalisation). This is optional — Voyage AI embeddings are fully compatible with HPC embeddings.

After deleting an episode

The episode_chunks rows are deleted automatically via ON DELETE CASCADE on the foreign key. No manual cleanup needed.

Chat System Prompt

Every chat request sends this three-part system prompt to Claude:

System Prompt Structure 
// Part 1 — Always present
You are the MAKEIT OR BREAKIT podcast assistant.

COMPANY KNOWLEDGE:
{companyKnowledge}  // full server/knowledge/company.ts string

// Part 2 — Retrieved from pgvector (dynamic per query)
RELEVANT CONTENT FROM OUR KNOWLEDGE BASE:
[Episode: "Investments" @ 05:13 — Startup quality]
Discussion focuses on the quantity and quality of startups...

[Company info]
MAKEIT OR BREAKIT is a Portuguese podcast about technology...

// Part 3 — Behaviour rules (always present)
Rules:
- Cite episode title and timestamp when referencing episode content
- Always respond in the same language the user writes in (EN or PT)
- Return ONLY valid JSON: { "message", "actions?", "sources?" }
- Never make up episode content
- If you don't know, say so clearly

Response Format

Claude always returns structured JSON. The server parses it and returns to the client. If parsing fails (Claude returned plain text), the server wraps it in {"message": responseText}.

JSON Response 
{
  "message": "At 72:18, the guest reveals a bold electric motorcycle venture...",

  "actions": [                      // optional — relevant links
    { "label": "Watch Episode", "href": "/podcasts/1?t=72:18" },
    { "label": "Apply as a Guest", "href": "/contact" }
  ],

  "sources": [                      // optional — citations
    {
      "episodeTitle": "Investments",
      "timeRef": "72:18",
      "topic": "Electric motorcycle project Portugal"
    }
  ]
}

11

Vector Search

SQL — searchChunks() 
SELECT ec.*, p.title AS episode_title,
  1 - (ec.embedding <=> $vector::vector) AS similarity
FROM episode_chunks ec
LEFT JOIN podcasts p ON p.id = ec.episode_id
WHERE 1 - (ec.embedding <=> $vector::vector) > 0.35
ORDER BY ec.embedding <=> $vector::vector
LIMIT 12
ParameterValueNotes
Distance metric <=> (cosine) Both models use L2-normalised vectors — cosine = dot product
Similarity threshold 0.35 Low — broad recall. Raise to 0.5 for more precise results
Top-K 12 Max chunks sent to Claude as context
Index type HNSW Approximate nearest-neighbour — fast even at scale

Troubleshooting

❌ "different vector dimensions 1024 and 1536"

The query embedding has a different dimension than stored embeddings. Check that getEmbedding() and generateAndStoreEmbeddings() in routes.ts both use voyage-3 (1024 dims), NOT voyage-large-2 (1536 dims).

Fix
sed -i "s/voyage-large-2/voyage-3/g" server/routes.ts
🤷 Chatbot says "I don't have information about that"

Either the topic isn't in any episode, or the similarity threshold is too high. Try:

1. Lower threshold from 0.35 to 0.25 in searchChunks()

2. Check that the topic is covered in episode key moments

3. Check that company knowledge covers this topic

4. Verify chunks exist: SELECT COUNT(*) FROM episode_chunks; in Neon SQL editor

💥 HPC job fails immediately (<10 seconds)

Run cat embed_*.err on MareNostrum. Common causes:

Python module not loaded: Run module load python/3.10.15-gcc before activating venv.

Model not found: Check path in embed_episodes.slurm points to ~/podcast-embed/models/bge-large-en-v1.5

Corrupted venv:

Fix
module load python/3.10.15-gcc
rm -rf ~/venvs/embed
python -m venv ~/venvs/embed
source ~/venvs/embed/bin/activate
pip install --no-index --find-links=offline_pkgs \
  sentence-transformers torch numpy ...
🌊 Voyage AI embeddings not appearing after adding episode

The embedding is fire-and-forget — check server logs for errors containing generateAndStoreEmbeddings. Most likely cause: invalid or missing VOYAGE_API_KEY.

Check Neon: SELECT * FROM episode_chunks WHERE episode_id = YOUR_ID;

🔤 Chat returning plain text instead of JSON

Claude occasionally returns plain text for complex queries. The server has a fallback that wraps it: {"message": responseText}. If it's causing 500 errors, check the JSON parsing in POST /api/chat in routes.ts — it should use a try/catch with fallback.
